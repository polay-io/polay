# Staking

POLAY uses Delegated Proof-of-Stake (DPoS) to secure the network. Validators lock POL tokens to participate in consensus and earn rewards. Delegators stake POL to validators they trust, increasing the validator's weight and sharing in the rewards.

## Validator Registration

Any address with sufficient POL can become a validator by submitting a `RegisterValidator` transaction:

```rust
RegisterValidator {
    self_stake: u64,            // initial POL stake (must meet minimum)
    commission_rate: u64,       // basis points (100 = 1%, max 10000 = 100%)
}
```

### Requirements

- **Minimum self-stake:** 100,000 POL (configurable chain parameter). This ensures validators have significant economic interest in the chain's security.
- **Commission rate:** Between 0 and 5,000 basis points (0% to 50%). Validators cannot set commission above the max to protect delegators from extractive operators.
- **One validator per address.** An address cannot register multiple validators.
- **Registration fee:** 50,000 base units of POL (paid in addition to the self-stake).

### What happens on registration

1. The `self_stake` amount is deducted from the sender's account balance and locked in the validator record.
2. A `ValidatorInfo` entry is created in state at key `0x07 ++ address`.
3. The validator is in `Registered` status but not yet in the active set.
4. At the next epoch boundary, if the validator's total stake places them in the top N validators, they enter the active set.

## Delegation

Any POL holder can delegate to a registered validator:

```rust
Delegate {
    validator: Address,         // the validator to delegate to
    amount: u64,                // POL to delegate
}
```

### Delegation mechanics

1. The `amount` is deducted from the delegator's account balance.
2. A `Delegation` entry is created or updated at key `0x08 ++ delegator ++ validator`.
3. The validator's `total_stake` is increased by `amount`.
4. The delegation takes effect immediately for reward accrual. The validator's consensus weight is updated at the next epoch boundary.

### Delegation limits

- **Minimum delegation:** 1,000 POL. Prevents dust delegations that add state bloat without meaningful economic weight.
- **No maximum per delegator.** A delegator can stake any amount to any number of validators.
- **No delegation cap per validator.** A validator can accept unlimited delegations. (A future governance parameter may cap this to prevent centralization.)

## Undelegation

Delegators can withdraw their stake by submitting an `Undelegate` transaction:

```rust
Undelegate {
    validator: Address,
    amount: u64,
}
```

### Unbonding period

Undelegated tokens are not immediately available. They enter an **unbonding period** of 14 epochs (configurable). During this period:

- The tokens are deducted from the validator's `total_stake` immediately (affecting their consensus weight at the next epoch boundary).
- The tokens do not earn rewards.
- The tokens cannot be transferred or re-delegated.
- An `UnbondingEntry` is created at key `0x0C ++ delegator ++ validator ++ epoch`.

After the unbonding period completes, the tokens are automatically returned to the delegator's account balance. In the MVP, this is processed at the epoch transition that follows the unbonding completion.

### Why unbonding exists

The unbonding period serves two purposes:

1. **Slashing accountability.** If a validator misbehaves, their delegators' tokens can be slashed. Without an unbonding period, a delegator could instantly withdraw before slashing is applied.
2. **Validator set stability.** Sudden large withdrawals could destabilize the validator set. The unbonding period smooths out stake changes over time.

## Commission

Validators earn commission on the rewards generated by their delegators' stake.

### How commission works

When rewards are distributed to a validator's stake pool:

1. The total reward is computed based on the validator's proportion of total active stake.
2. The validator's commission (e.g., 10% = 1000 basis points) is deducted first and credited to the validator's account.
3. The remaining reward is distributed proportionally among all delegators (including the validator's self-stake).

### Example

Validator V has:
- Self-stake: 100,000 POL
- Delegations: 400,000 POL from various delegators
- Commission: 10% (1000 basis points)
- Total stake: 500,000 POL

Block reward allocated to V: 5,000 POL

1. Commission: 5,000 * 10% = 500 POL -> validator's account
2. Remaining: 4,500 POL
3. Validator's self-stake share: 4,500 * (100,000 / 500,000) = 900 POL -> validator's account
4. Delegator A (200,000 POL): 4,500 * (200,000 / 500,000) = 1,800 POL -> A's reward balance
5. Delegator B (200,000 POL): 4,500 * (200,000 / 500,000) = 1,800 POL -> B's reward balance

Total validator earnings: 500 + 900 = 1,400 POL (28% of rewards despite having 20% of stake).

### Commission changes

A future upgrade will allow validators to change their commission rate with a notice period (e.g., commission changes take effect after 2 epochs). This prevents validators from luring delegators with low commission and then suddenly increasing it.

## Reward Distribution

### Sources of rewards

| Source | Distribution |
|---|---|
| **Block rewards** | Fixed amount of POL minted per block (inflationary). Distributed to the active validator set proportional to stake. |
| **Transaction fees** | 70% to the block proposer, 30% to the protocol treasury. The proposer's share is further split between the validator and their delegators based on commission. |

### Block reward schedule

The MVP uses a fixed block reward of 10 POL per block. A decaying reward schedule (halving every N epochs) is planned for mainnet to control inflation.

### Claiming rewards

Rewards accrue in a virtual balance tracked per delegation. Delegators must explicitly claim rewards by submitting a `ClaimRewards` transaction:

```rust
ClaimRewards {
    validator: Address,         // claim rewards from this validator
}
```

This transfers all accrued rewards from the delegation's reward balance to the delegator's account balance. Claimed rewards are then freely transferable.

### Reward accounting

Reward distribution uses a **cumulative reward per share** model (similar to Cosmos SDK's F1 fee distribution):

1. Each validator tracks a `cumulative_reward_ratio` that increases each time rewards are added.
2. Each delegation tracks a `reward_debt` set at the time of delegation or last claim.
3. Claimable rewards = `delegation.amount * (validator.cumulative_reward_ratio - delegation.reward_debt)`.

This allows O(1) reward calculation per claim without iterating over all historical reward distributions.

## Slashing

Slashing penalizes validators and their delegators for behavior that threatens consensus safety or liveness.

### Double signing

A validator signs two conflicting consensus messages (blocks, prevotes, or precommits) at the same height and round.

**Penalty:**
- **5% of total stake** (self-stake + delegations) is burned. Burned tokens are permanently removed from supply.
- The validator is **jailed** for 7 days (configurable).
- Delegators are slashed proportionally -- if a delegator has 10% of the validator's delegations, they lose 10% of the slashed amount from their delegation.

### Downtime

A validator fails to participate in consensus for more than 50 consecutive blocks.

**Penalty:**
- **0.1% of total stake** is burned.
- The validator is jailed for 1 hour.

### Jailing

When a validator is jailed:

1. They are immediately removed from the active validator set.
2. They cannot propose blocks or vote in consensus.
3. Their delegators' tokens remain staked but do not earn rewards.
4. After the jail period expires, the validator can submit an `Unjail` transaction to rejoin.

### Tombstoning

A validator that has been jailed 3 times is **tombstoned**: permanently removed from the validator set. Their self-stake is returned (minus slashing penalties). Delegators can undelegate immediately (unbonding period is waived for tombstoned validators) without further slashing risk.

Tombstoning is irreversible. The address cannot register as a validator again. This removes persistently misbehaving validators from the system.

## Validator Lifecycle

```
Account with sufficient POL
        |
        | RegisterValidator
        v
  [ Registered ]
        |
        | (top N by stake at epoch boundary)
        v
  [ Active ] -----------> [ Jailed ]
        ^                    |    |
        |                    |    | (3rd offense)
        | Unjail (after      |    v
        | cooldown)          | [ Tombstoned ]
        |____________________|
```

### Status transitions

| From | To | Trigger |
|---|---|---|
| Registered | Active | Epoch boundary, validator is in top N by total stake |
| Active | Jailed | Double signing detected OR downtime threshold exceeded |
| Jailed | Active | Jail period expires AND validator submits Unjail AND still in top N |
| Jailed | Tombstoned | Third jailing offense |
| Active | Registered | Falls below top N by total stake at epoch boundary |
| Tombstoned | (terminal) | Cannot recover |

## Epoch-Based Updates

The active validator set is only updated at epoch boundaries (every 100 blocks by default). This provides:

- **Predictability:** Game studios and infrastructure providers know the validator set for the next 100 blocks. No mid-epoch surprises.
- **Batch processing:** Validator set changes, unbonding completions, and reward distributions are processed together, amortizing the cost.
- **Consensus stability:** The BFT protocol does not need to handle validator set changes during a round.

### Epoch transition steps

At block height `H` where `H % epoch_length == 0`:

1. Process completed unbonding entries (return tokens to delegators).
2. Distribute epoch rewards to validators and delegators.
3. Process slashing evidence accumulated during the epoch.
4. Compute the new active validator set (top N by total stake, excluding jailed/tombstoned).
5. Update the `active_validator_set` in chain metadata.
6. Increment the epoch counter.

## Chain Parameters

| Parameter | Default | Description |
|---|---|---|
| `min_self_stake` | 100,000 POL | Minimum self-stake to register as a validator |
| `min_delegation` | 1,000 POL | Minimum delegation amount |
| `max_validators` | 21 | Maximum active validator set size |
| `epoch_length` | 100 blocks | Blocks per epoch |
| `unbonding_period` | 14 epochs | Epochs before undelegated tokens are available |
| `max_commission` | 5,000 bps (50%) | Maximum commission rate |
| `block_reward` | 10 POL | POL minted per block |
| `double_sign_slash` | 500 bps (5%) | Slash percentage for double signing |
| `downtime_slash` | 10 bps (0.1%) | Slash percentage for downtime |
| `downtime_threshold` | 50 blocks | Consecutive missed blocks before downtime slashing |
| `jail_duration_double_sign` | 7 days | Jail period for double signing |
| `jail_duration_downtime` | 1 hour | Jail period for downtime |
| `max_jailings` | 3 | Jailings before tombstoning |

These parameters are set at genesis and can be modified via governance (future).
